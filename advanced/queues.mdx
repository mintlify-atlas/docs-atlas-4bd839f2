## Overview

EduStack Smart uses Laravel's queue system to handle time-consuming tasks asynchronously, improving application responsiveness and user experience. Tasks like media processing, email sending, and data exports are processed in the background.

## Queue Configuration

The queue system is configured in `config/queue.php`:

```php config/queue.php
return [
    'default' => env('QUEUE_CONNECTION', 'database'),
    
    'connections' => [
        'sync' => [
            'driver' => 'sync',
        ],
        
        'database' => [
            'driver' => 'database',
            'connection' => env('DB_QUEUE_CONNECTION'),
            'table' => env('DB_QUEUE_TABLE', 'jobs'),
            'queue' => env('DB_QUEUE', 'default'),
            'retry_after' => (int) env('DB_QUEUE_RETRY_AFTER', 90),
            'after_commit' => false,
        ],
        
        'redis' => [
            'driver' => 'redis',
            'connection' => env('REDIS_QUEUE_CONNECTION', 'default'),
            'queue' => env('REDIS_QUEUE', 'default'),
            'retry_after' => (int) env('REDIS_QUEUE_RETRY_AFTER', 90),
            'block_for' => null,
            'after_commit' => false,
        ],
        
        'sqs' => [
            'driver' => 'sqs',
            'key' => env('AWS_ACCESS_KEY_ID'),
            'secret' => env('AWS_SECRET_ACCESS_KEY'),
            'prefix' => env('SQS_PREFIX'),
            'queue' => env('SQS_QUEUE', 'default'),
            'region' => env('AWS_DEFAULT_REGION', 'us-east-1'),
            'after_commit' => false,
        ],
    ],
    
    'failed' => [
        'driver' => env('QUEUE_FAILED_DRIVER', 'database-uuids'),
        'database' => env('DB_CONNECTION', 'sqlite'),
        'table' => 'failed_jobs',
    ],
];
```

### Environment Variables

```bash .env
QUEUE_CONNECTION=database
DB_QUEUE_TABLE=jobs
DB_QUEUE=default
DB_QUEUE_RETRY_AFTER=90
```

## Queue Drivers

<Tabs>
  <Tab title="Database">
    **Best for:** Development and small applications
    
    ```bash
    QUEUE_CONNECTION=database
    ```
    
    Jobs are stored in the `jobs` database table. Simple to set up, no additional infrastructure required.
    
    **Pros:**
    - No additional setup
    - Easy to debug
    - Works on shared hosting
    
    **Cons:**
    - Slower than Redis
    - Can't handle high throughput
  </Tab>
  
  <Tab title="Redis">
    **Best for:** Production applications
    
    ```bash
    QUEUE_CONNECTION=redis
    REDIS_HOST=127.0.0.1
    REDIS_PASSWORD=null
    REDIS_PORT=6379
    ```
    
    Jobs are stored in Redis, providing fast, reliable queue processing.
    
    **Pros:**
    - Very fast
    - Handles high throughput
    - Supports multiple queues
    
    **Cons:**
    - Requires Redis server
    - More complex setup
  </Tab>
  
  <Tab title="SQS">
    **Best for:** AWS deployments
    
    ```bash
    QUEUE_CONNECTION=sqs
    AWS_ACCESS_KEY_ID=your-key
    AWS_SECRET_ACCESS_KEY=your-secret
    SQS_PREFIX=https://sqs.us-east-1.amazonaws.com/your-account
    SQS_QUEUE=default
    ```
    
    Jobs are stored in Amazon SQS, ideal for cloud-native applications.
    
    **Pros:**
    - Fully managed
    - Highly scalable
    - Integrates with AWS ecosystem
    
    **Cons:**
    - Costs money
    - Requires AWS account
  </Tab>
</Tabs>

## Media Processing Queue

EduStack Smart uses a dedicated queue for media processing to prevent blocking other tasks.

### Configuration

Media processing queue is configured in `config/media-library.php`:

```php config/media-library.php
return [
    // Queue connection for media processing
    'queue_connection_name' => env('QUEUE_CONNECTION', 'sync'),
    
    // Dedicated queue name for media jobs
    'queue_name' => env('MEDIA_QUEUE', 'media_processing'),
    
    // Queue conversions by default
    'queue_conversions_by_default' => env('QUEUE_CONVERSIONS_BY_DEFAULT', true),
    
    // Wait for database commit before queueing
    'queue_conversions_after_database_commit' => env('QUEUE_CONVERSIONS_AFTER_DB_COMMIT', true),
    
    // Job classes
    'jobs' => [
        'perform_conversions' => Spatie\MediaLibrary\Conversions\Jobs\PerformConversionsJob::class,
        'generate_responsive_images' => Spatie\MediaLibrary\ResponsiveImages\Jobs\GenerateResponsiveImagesJob::class,
    ],
];
```

### How Media Processing Works

<Steps>
  <Step title="Upload Media">
    User uploads an image through the web interface.
    
    ```php
    $post->addMediaFromRequest('image')
        ->toMediaCollection('gallery');
    ```
  </Step>
  
  <Step title="Queue Jobs">
    Spatie Media Library automatically queues conversion jobs:
    - `PerformConversionsJob` - Creates image variants (thumbnails, hero images, etc.)
    - `GenerateResponsiveImagesJob` - Generates responsive image sizes
  </Step>
  
  <Step title="Process Jobs">
    Queue worker processes jobs in the background:
    
    ```bash
    php artisan queue:work --queue=media_processing
    ```
  </Step>
  
  <Step title="Broadcast Completion">
    After processing, `MediaProcessed` event is broadcast to the client:
    
    ```php
    event(new MediaProcessed($post->id, 'post'));
    ```
  </Step>
  
  <Step title="Client Updates">
    React component receives the event and reloads data:
    
    ```typescript
    const echo = useEcho(`post.${postId}`, 'MediaProcessed', () => {
        router.reload({ only: ['post'] });
    });
    ```
  </Step>
</Steps>

## Running Queue Workers

### Development

For development, use `queue:listen` which automatically reloads code changes:

```bash
php artisan queue:listen --tries=1
```

The project includes a convenient dev script:

```bash
composer dev
```

This runs:
- Laravel development server
- Queue worker with auto-reload
- Vite development server

### Production

In production, use `queue:work` with a process manager like Supervisor:

```bash
php artisan queue:work --sleep=3 --tries=3 --max-time=3600
```

#### Supervisor Configuration

```ini /etc/supervisor/conf.d/edustack-worker.conf
[program:edustack-worker]
process_name=%(program_name)s_%(process_num)02d
command=php /var/www/edustack/artisan queue:work database --sleep=3 --tries=3 --max-time=3600
autostart=true
autorestart=true
stopasgroup=true
killasgroup=true
user=www-data
numprocs=4
redirect_stderr=true
stdout_logfile=/var/www/edustack/storage/logs/worker.log
stopwaitsecs=3600
```

<Info>
Run `numprocs=4` workers for better throughput. Adjust based on your server's CPU cores.
</Info>

#### Managing Supervisor

```bash
# Reload configuration
sudo supervisorctl reread
sudo supervisorctl update

# Start workers
sudo supervisorctl start edustack-worker:*

# Stop workers
sudo supervisorctl stop edustack-worker:*

# Restart workers (after deployment)
sudo supervisorctl restart edustack-worker:*

# Check status
sudo supervisorctl status
```

## Creating Custom Jobs

Create a new job using Artisan:

```bash
php artisan make:job ProcessVideoUpload
```

Implement the job:

```php app/Jobs/ProcessVideoUpload.php
namespace App\Jobs;

use App\Models\Post;
use Illuminate\Bus\Queueable;
use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Foundation\Bus\Dispatchable;
use Illuminate\Queue\InteractsWithQueue;
use Illuminate\Queue\SerializesModels;

class ProcessVideoUpload implements ShouldQueue
{
    use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;
    
    public $timeout = 600; // 10 minutes
    public $tries = 3;
    public $maxExceptions = 2;

    public function __construct(
        public Post $post,
        public string $videoPath
    ) {}

    public function handle(): void
    {
        // Process video
        // Generate thumbnail
        // Convert to different formats
        // Update database
        
        event(new MediaProcessed($this->post->id, 'post'));
    }
    
    public function failed(\Throwable $exception): void
    {
        // Send notification about failure
        // Log error
        // Clean up temporary files
    }
}
```

Dispatch the job:

```php
use App\Jobs\ProcessVideoUpload;

// Dispatch immediately
ProcessVideoUpload::dispatch($post, $videoPath);

// Dispatch with delay
ProcessVideoUpload::dispatch($post, $videoPath)->delay(now()->addMinutes(5));

// Dispatch to specific queue
ProcessVideoUpload::dispatch($post, $videoPath)->onQueue('videos');
```

## Queue Priorities

Run workers with multiple queues in priority order:

```bash
php artisan queue:work --queue=high,default,low
```

Dispatch jobs to specific queues:

```php
// High priority
ProcessVideoUpload::dispatch($post, $videoPath)->onQueue('high');

// Default priority
ProcessVideoUpload::dispatch($post, $videoPath);

// Low priority
ProcessVideoUpload::dispatch($post, $videoPath)->onQueue('low');
```

## Monitoring Queues

### Artisan Commands

```bash
# View jobs in queue
php artisan queue:monitor database

# View failed jobs
php artisan queue:failed

# Retry failed job
php artisan queue:retry {id}

# Retry all failed jobs
php artisan queue:retry all

# Delete failed job
php artisan queue:forget {id}

# Clear all failed jobs
php artisan queue:flush
```

### Laravel Horizon (Optional)

For advanced queue monitoring, consider installing Laravel Horizon:

```bash
composer require laravel/horizon
php artisan horizon:install
```

Horizon provides a beautiful dashboard for monitoring queues, jobs, and worker metrics.

## Batch Processing

Process multiple jobs as a batch:

```php
use Illuminate\Bus\Batch;
use Illuminate\Support\Facades\Bus;
use App\Jobs\ProcessImage;

$batch = Bus::batch([
    new ProcessImage($image1),
    new ProcessImage($image2),
    new ProcessImage($image3),
])->then(function (Batch $batch) {
    // All jobs completed successfully
    event(new BatchProcessingComplete($batch->id));
})->catch(function (Batch $batch, Throwable $e) {
    // First batch job failure detected
})->finally(function (Batch $batch) {
    // The batch has finished executing
})->dispatch();

return $batch->id;
```

Check batch status:

```php
$batch = Bus::findBatch($batchId);

if ($batch->finished()) {
    // All jobs completed
}

if ($batch->cancelled()) {
    // Batch was cancelled
}

// Progress: 0-100
$progress = $batch->progress();
```

## Testing Jobs

Test jobs without actually dispatching them:

```php
use Illuminate\Support\Facades\Queue;
use App\Jobs\ProcessVideoUpload;

Queue::fake();

// Perform action that dispatches job
$response = $this->post('/videos', ['video' => $file]);

// Assert job was dispatched
Queue::assertPushed(ProcessVideoUpload::class);

// Assert job was pushed to specific queue
Queue::assertPushedOn('videos', ProcessVideoUpload::class);

// Assert job was not dispatched
Queue::assertNotPushed(AnotherJob::class);
```

## Best Practices

<AccordionGroup>
  <Accordion title="Keep Jobs Small">
    Break large tasks into smaller jobs for better reliability and progress tracking.
    
    ```php
    // Bad: One large job
    ProcessEntireDataset::dispatch($dataset);
    
    // Good: Multiple smaller jobs
    foreach ($dataset->chunks(100) as $chunk) {
        ProcessDataChunk::dispatch($chunk);
    }
    ```
  </Accordion>

  <Accordion title="Set Timeouts">
    Always set appropriate timeouts to prevent jobs from running indefinitely.
    
    ```php
    public $timeout = 300; // 5 minutes
    ```
  </Accordion>

  <Accordion title="Handle Failures">
    Implement the `failed()` method to clean up after job failures.
    
    ```php
    public function failed(Throwable $exception): void
    {
        // Clean up resources
        // Notify admin
        // Log error
    }
    ```
  </Accordion>

  <Accordion title="Use Queue-Specific Connections">
    Separate heavy jobs from lightweight jobs using different queues.
    
    ```bash
    # Terminal 1: Process media
    php artisan queue:work --queue=media_processing
    
    # Terminal 2: Process everything else
    php artisan queue:work --queue=default
    ```
  </Accordion>
</AccordionGroup>
